### Refactor notes
REFACTOR HAZARDS: (1) Hoisting/init-order: colorGrid allocation at line 370 calls getEmptyColor() defined at line 547, and the bakeBackground IIFE (623) calls valueNoise/hash2 — all rely on function-declaration hoisting within one script; splitting into ES modules must preserve definition-before-use or these become TDZ/undefined errors. (2) Forward references to later code: updateCamera reads `player` and `keys`; updateFlyingParticles calls updateScoreUI/damagePlayer and reads `player`; triggerExplosion calls damageEnemy/damagePlayer/cellBlocks and reads `enemies`/`player`/`gameMode` — all declared AFTER this range; safe only because they execute at runtime. Circular-import risk when modularizing (explosion <-> entities, particles <-> player/UI). (3) Shared mutable const objects: cam, sim, GLOBAL_PARAMS, MATERIAL_PARAMS, SPELL_PARAMS, sfxThrottle, mouseGridPosition are mutated in place across systems; modules must share the same object reference (or move to setters). UI inspector code mutates MATERIAL_PARAMS/SPELL_PARAMS fields directly. (4) Four-grid lockstep invariant: grid/colorGrid/lifeGrid/chargeGrid must always move together (swapCells is the only safe primitive); movedGrid is set-true-on-both, never swapped, and is cleared per tick elsewhere — an ordering dependency with the CA pass. (5) Color aliasing: colorGrid entries are [r,g,b] array references; spawnFlyingParticle/triggerExplosion capture them into particles and later re-deposit the SAME array into the grid; stainCell replaces (not mutates) entries. Making colors value types changes behavior subtly. (6) Frame-order dependency: updateCamera must run before updateSimBounds (sim derives from cam); sim bounds gate which cells the CA simulates (M=44 margin), so anything outside the camera window is frozen — moving the camera 'wakes' regions. renderCamX/renderCamY are a separate integer snapshot taken at render time. (7) Real-time vs sim-time: sfxThrottle uses performance.now; playCoin/playWaveHorn schedule second notes with setTimeout — not paused by simSpeed=0. Audio context creation must happen inside a user gesture (ensureAudio called from input handlers elsewhere). (8) Module-load side effects: renderer creation + holder.appendChild + background bake run at import time; in TS, isolate into an init() to keep modules importable in tests. (9) Magic constants that matter: camera lerp 0.085, zoom lerp 0.035, idle threshold 55 frames, zoom 1.13, lookahead 26px, build-mode pan 9px, sim margin 44, MAX_PARTICLES 4200 (silent drop), default particle grav 0.16, shockwave {speed 3.5, strength 12, maxRadius radius*2.2}, explosion FIRE 30%/SMOKE 20% conversion, debris launch 22%, WALL rim survival 45% beyond 0.55*r^2, charge=4 at 40%, enemy damage radius radius*1.6 vs player radius*1.5 (player dmg cap 42), bloomKick cap 0.95, screenShake cap 0.045, quad overscan 4/VIEW px, noise seed 1337. (10) triggerExplosion is the cross-cutting choke point: it touches grids, particles, shockwaves, bloom/shake render state, audio, enemies, and player — plan its module's dependency direction carefully (likely event/callback-based to break cycles). (11) updateFlyingParticles iterates backward with splice — preserve iteration order when porting. (12) Grids are column-major (grid[x][y]) JS arrays of arrays; a typed-array flattening refactor must touch every [x][y] access including swapCells and explosion loops.
